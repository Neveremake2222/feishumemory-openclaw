# 最终改进

> 本文档基于当前仓库文档、benchmark 报告和核心代码扫描整理，用于后续逐步修改。原则是不削弱 benchmark 断言，不为了数字好看而降低测试标准，失败 case 应继续暴露真实引擎缺陷。

## 总体判断

项目已经具备可运行 MVP，包括结构化记忆写入、召回、飞书接入、OpenClaw 适配、Dashboard 演示、多 agent 治理投票雏形和 benchmark 框架。

当前主要问题不是“完全没有功能”，而是：

- 报告版本不一致，需要重新生成当前代码版本的统一评测报告。
- 关键 benchmark 暴露的零结果拒绝、偏好召回、决策版本链仍需作为真实缺陷修复。
- 生产化能力还不完整，包括权限、日志脱敏、真实飞书 source resolver、并发安全、产品壳完善。
- 后续优化应优先围绕真实失败 case，而不是继续增加演示性功能。

## P0：必须优先完成

### 1. 重新生成当前版本的统一评测报告

现状：

- `BENCHMARK_REPORT.md` 显示 5,348 case，总通过率 80.3%。
- `docs/pending-optimizations.md` 又记录 345 tests、71/71 benchmarks passed。
- 多份文档对应不同时间点，不能直接作为最终答辩或产品说明依据。

要做：

- 重新运行当前代码版本的完整 benchmark。
- 输出唯一可信报告，包含总 case、各 Track 通过率、失败类型、P50/P95、baseline 对比。
- 明确标注运行时间、代码版本、数据集版本、运行参数。

验收标准：

- 生成新的当前版本报告。
- 报告中的 Track、case 数、通过率和 benchmark 目录保持一致。
- 失败 case 不被删除、不被弱化。

### 2. 修复零结果拒绝问题

现状：

- 报告中 `zero_result` 为 0/500。
- 引擎对不存在的话题仍返回记忆，属于幻觉风险。

要做：

- 增加最小相关性阈值。
- 判断 BM25 是否有真实 query term 命中。
- 对低于阈值的候选返回空结果。
- 在 API 和 Dashboard 中明确展示“没有可靠记忆”，而不是硬凑结果。

验收标准：

- zero-result case 必须返回 0 条结果。
- recall_log 记录零结果占位。
- 不影响已有高相关 query 的正常召回。

### 3. 修复偏好召回问题

现状：

- 报告中 `preference_recall` 为 0/150。
- 代码已有 `implicit_preferences.py` 和多 agent 治理投票，但需要确认 benchmark 下是否真正有效。

要做：

- 检查 preference 写入是否有稳定结构化字段。
- 增加 preference 专用 ranking boost。
- 明确旧偏好 superseded 过滤规则。
- 强化 user scope 隔离。
- 对 conflict preference 增加可解释返回。

验收标准：

- 当前偏好优先于旧偏好。
- 用户 A 的 private preference 不被用户 B 召回。
- preference recall benchmark 有真实改善。
- 失败 case 继续保留为回归测试。

### 4. 修复决策版本链问题

现状：

- 报告中 `decision_version` 为 25%。
- 已 superseded 的旧决策可能因为关键词重合被召回。

要做：

- 增加 currentness gate。
- 普通召回默认排除 superseded decision。
- 用户明确询问“历史版本/之前的方案/变更过程”时才允许返回版本链。
- 在结果中清楚区分当前决策和历史决策。

验收标准：

- 当前决策 query 只返回最新 active decision。
- 历史版本 query 可以返回版本链。
- 已弃用决策不会压过当前决策。

### 5. 分析并提升 Track L Agent Task

现状：

- Track L 为 48/150，32%。
- 这是端到端 agent task 能力缺口，不只是单纯检索问题。

要做：

- 对失败 case 分类：没召回、召回太多、引用错误、任务链断裂、回答层未正确使用记忆。
- 固化失败 case 到 `benchmarks/failure_cases/`。
- 修复后作为回归门禁。

验收标准：

- 每个失败类型有明确原因。
- 至少保留一批代表性失败 case。
- 修复后 Track L 指标提升，但不降低评分标准。

## P1：高价值工程补齐

### 1. 实现真实飞书 source resolver

现状：

- `validate_sources()` 有接口，但真实飞书消息/文档 resolver 仍不完整。

要做：

- 按 `source_ref/source_version/content_hash` 校验飞书来源。
- 判断来源是否存在、是否变更、是否删除。
- 单条 source 校验失败不能中断整批校验。

验收标准：

- `ok / changed / missing / unknown` 状态可靠返回。
- source validation 保持只读，不自动误删记忆。

### 2. 完善 Dashboard 产品壳

现状：

- 当前更接近本地演示台，不是完整 SaaS 产品。

要做：

- 增加真实数据刷新状态。
- 增加飞书长连接状态展示。
- 增加空状态、错误态、加载态。
- 优化记忆卡片展示和摘要层级。

验收标准：

- 使用本地数据库和飞书长连接时能稳定展示。
- 查询结果默认显示摘要，点击卡片后再展示出处、时间和原记录。

### 3. 处理 SQLite 并发风险

现状：

- `MemoryEngine` 使用共享 SQLite connection 和 `check_same_thread=False`。
- FastAPI 多请求下可能存在事务交错或锁竞争。

要做：

- MVP 阶段先加 `RLock` 串行化 write、recall、update、compact、review。
- 后续改为 per-request connection 或队列化写入。

验收标准：

- 并发 write/recall smoke test 不报 SQLite 线程或事务错误。
- 原有 API 测试继续通过。

### 4. 加强隐私和日志脱敏

现状：

- 飞书 daemon、Dashboard、API 日志需要继续确认是否泄露原文和 secret。

要做：

- 日志中不打印原始消息正文。
- 打印内容前调用脱敏逻辑。
- `.env`、API key、Bearer token 不进入日志、截图、报告。

验收标准：

- 日志中不出现手机号、邮箱、token、API key。
- ingest 正常写入不受影响。

### 5. 明确权限和 visibility 模型

现状：

- 已有 `user_id/project_id/task_id/scope`，但缺少更清楚的可见性字段。

要做：

- 增加或模拟 `visibility = private/project/org`。
- private preference 默认只对本人可见。
- project decision 对同项目可见。

验收标准：

- 用户 A 的 private preference 不会被用户 B 召回。
- 同项目成员能召回 project decision。
- benchmark 或单元测试覆盖跨用户隔离。

### 6. 增加结构化 facts 层

现状：

- memories 中有结构化 content，但没有独立 facts 表。

要做：

- 新增 `facts` 表，最小字段包括 subject、predicate、object、scope、project_id、task_id、user_id、source_memory_id、confidence、status。
- 初期只从明确 decision/preference 抽取 facts。
- recall 时可返回 `facts_summary`。

验收标准：

- “当前方案是什么”“当前负责人是谁”“默认规范是什么”可以从 facts 层直接回答。
- fact 可回溯到 source memory。
- decision supersede 时同步更新 fact 状态。

### 7. 增加时间有效性

现状：

- task_status、decision、preference 缺少统一的有效期模型。

要做：

- 增加 `valid_from / valid_until`。
- task_status 默认短期有效。
- decision/preference 默认长期有效。
- 过期状态不再优先召回。

验收标准：

- 过期 task_status 不压过当前状态。
- 当前决策仍长期可用。
- Dashboard 能区分当前有效记忆和历史记忆。

### 8. 稳定后台维护调度

现状：

- `heartbeat.py` 已有基础，但还需要稳定 scheduler。

要做：

- 定期执行 `compact/review/validate_sources`。
- 提供环境变量开关。
- 记录维护日志和审计信息。
- benchmark 模式下可关闭，保证实验可复现。

验收标准：

- 后台维护不影响正常 API recall/write。
- 可手动或定时触发。

## 2026-05-06 时间有效性实施记录

### P1 valid_from / valid_until 最小闭环

已完成：

- `memories` 表新增 nullable 字段：
  - `valid_from`
  - `valid_until`
- `insert_memory()` 会从 `MemoryCandidate.content` 中读取 `valid_from / valid_until` 并写入独立列。
- recall 过滤层增加时间有效性判断：
  - 当前时间早于 `valid_from`：不返回。
  - 当前时间大于等于 `valid_until`：不返回。
  - 未设置有效期：保持原有行为，仍可召回。
- 兼容已有数据：老数据没有 `valid_from / valid_until` 时不受影响。
- working memory promote 到 factual 时保留 `valid_from / valid_until`。

当前设计边界：

- 本轮只做召回期过滤，不自动 archive 过期 memory。
- `valid_from / valid_until` 先由写入方或抽取器显式提供，尚未实现按 `task_status` 自动推导短期有效期。
- Dashboard 还未展示“当前有效/历史过期”的视觉区分。

验证结果：

- 新增测试：
  - 过期 memory 不召回。
  - 未来才生效 memory 不召回。
  - 当前有效窗口内 memory 正常召回，并验证字段入库。

### P1 task_status 默认短期有效期

已完成：

- `task_status` 在写入时如果没有显式提供 `valid_until`，会自动写入默认有效期。
- 默认有效期为 14 天，和现有 freshness decay 口径保持一致，避免 7 天 TTL 误伤仍应召回的 8 天内状态类 benchmark case。
- 显式 `valid_until` 优先，写入方提供的时间不会被默认值覆盖。
- 支持写入方通过 `ttl_hours` 或 `validity_days` 自定义状态类记忆的有效期。
- 非 `task_status` 类型不受影响；未设置有效期的老数据继续保持原有行为。

设计边界：

- 默认有效期基于写入时间计算，不基于源消息时间回填。这样可以避免导入历史消息或 benchmark 回写 `created_at` 时被立即判定过期。
- 本轮仍只在 recall 阶段过滤过期记忆，不自动 archive 过期记忆。
- 后续如果需要更严格的项目管理语义，可以由飞书抽取器根据“延期到/截止到/已完成/已取消”等语言写入更精确的 `valid_until` 或 `valid_from`。

  - 显式 `valid_until` 不被默认 TTL 覆盖。
  - `ttl_hours` 能控制自定义有效期。

### P0 全量 benchmark 分片汇总报告

已完成：

- `benchmarks.report` 新增 `--sharded` 模式。
- `benchmarks.report` 新增 `--shards` 参数，支持用分号声明多个 shard，例如：

```powershell
python -m benchmarks.report benchmarks_runtime\benchmark_report_sharded_current.md --no-baselines --sharded
python -m benchmarks.report benchmarks_runtime\benchmark_report_sharded_AB_current.md --no-baselines --shards 'A;B'
python -m benchmarks.report benchmarks_runtime\benchmark_report_sharded_custom.md --no-baselines --shards 'A,B;C;J,J-gen;K;L;M'
```

- 默认 `--sharded` 会按 Track 逐个分片运行，降低完整 suite 单次运行超时后丢失全部结果的风险。
- 每个 shard 会写出独立 Markdown 报告，路径位于聚合报告旁边的 `<report_name>_shards/` 目录。
- 聚合报告顶部新增 `Shard Manifest`，展示：
  - shard 序号
  - 包含 Track
  - passed/total/failed
  - 对应 shard 报告路径
- 聚合报告下半部分继续复用原有 `Benchmark Report` 结构，包含：
  - Track 通过率
  - case count
  - dataset readiness
  - baseline comparison
  - failure type distribution
  - memory tax summary

使用注意：

- PowerShell 中 `--shards` 的分号必须加引号，例如 `--shards 'A;B'`，否则分号会被 PowerShell 当成命令分隔符。
- `--sharded/--shards` 不能和 `--critical` 或 `--ablation` 混用。
- 如果需要 baseline 对比，去掉 `--no-baselines` 即可，但运行时间会按 baseline mode 成倍增加。
